## Cómo se obtiene y valida la sesión

1. Al cargar el POS, `ProtectedRoute` conserva sólo el token como credencial (el parámetro legacy `user` se elimina de la URL).
2. `PosSessionProvider` llama a `GET /api/v1/auth/session` usando `fetchApi`.
3. La respuesta (`user`, `roles`, `permissions`, `scope`, `active_branch`) se almacena en el contexto de React.
4. `hasPermission(code)` consulta los permisos de la sesión canónica — nunca de `localStorage`.
5. Para `scope.level == "branch"`, el `active_branch.id` reemplaza cualquier `branch_id` local; el Supervisor no tiene selector de sucursal.
6. Errores: 401 → limpia tokens y redirige a `/admin/login`; 403 → pantalla de acceso denegado; red/503 → error recuperable con botón Reintentar.

## Comportamiento de Supervisor y Cajero

- **Supervisor:** ve el menú Administración si tiene `branch.admin.access`; puede consultar productos, modificar disponibilidad con `catalog.branch.manage`, consultar personal con `branch.staff.read`, y ver su sucursal activa. No puede modificar catálogos centrales, usuarios, roles ni sucursales.
- **Cajero:** no ve el menú Administración; si escribe `/pos/administration` directamente, `PermissionRoute` lo redirige a `/pos/pos`.

## Contratos consumidos

- `GET /api/v1/auth/session` — sesión canónica.
- `GET /api/v1/branch-administration/context` — sucursal activa.
- `GET /api/v1/branch-administration/staff` — personal.
- `GET /api/v1/branch-administration/catalog/products` — productos.
- `PUT /api/v1/branch-administration/catalog/products/{id}/availability` — disponibilidad (`available`, `unavailable`, `inherit`).

## Riesgos y desviaciones

- **Validación visual:** el entorno no permite validación visual en navegador. El typecheck y build exitosos confirman que el código compila y empaqueta; la validación visual queda pendiente para el despliegue a staging.
- **React Query disponible pero no usado:** los componentes nuevos usan `useEffect`+`fetchApi` (patrón existente del POS) en lugar de `useQuery`, para mantener consistencia con el resto del POS. Una migración a React Query puede hacerse en un incremento futuro.
- **`setPosBranchId` legacy:** se conserva para no romper componentes existentes no tocados por BA-002. La sesión canónica (`PosSessionProvider`) es la fuente para BA-002; el `branch_id` que llega a las operaciones de caja sigue siendo validado por el backend.
