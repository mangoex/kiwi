## Diagnóstico

Al ejecutar `alembic upgrade head` sobre la base PostgreSQL de Easypanel (detenida en `0013_pos_cash_rbac_permissions`), la transacción se revierte antes de registrar la revisión `0014_legacy_caja_role_permissions`:

```text
StringDataRightTruncation:
value too long for type character varying(32)

UPDATE alembic_version
SET version_num='0014_legacy_caja_role_permissions'
```

La tabla técnica `alembic_version.version_num` es `VARCHAR(32)`, pero los identificadores de revisión 0014 a 0018 miden entre 33 y 37 caracteres. La transacción se revierte y la base permanece en `0013_pos_cash_rbac_permissions`.

## Causa

La longitud del identificador de revisión creció con los nombres descriptivos usados desde la revisión 0010, superando los 32 caracteres del `version_num` estándar de Alembic a partir de la 0014. Las revisiones afectadas:

- `0014_legacy_caja_role_permissions` — 33 caracteres
- `0015_business_units_operational_roles` — 37 caracteres
- `0016_customer_directory_snapshots` — 33 caracteres
- `0017_suppliers_purchase_presentations` — 37 caracteres
- `0018_direct_purchases_cash_costing` — 34 caracteres

`0019_advanced_recipes_production` (32) está justo en el límite y cabía. Las revisiones previas a 0014 no superaban 30 caracteres.

## Estado productivo observado en 0013

La base PostgreSQL de Easypanel se encuentra sana en `0013_pos_cash_rbac_permissions` (30 caracteres). La migración fallida fue revertida por PostgreSQL (DDL transaccional), de modo que no quedó estado parcial. El esquema y los datos de negocio están intactos en ese punto.

## Diseño de la migración puente

Se inserta una migración puente entre `0013_pos_cash_rbac_permissions` y `0014_legacy_caja_role_permissions`:

```
0013_pos_cash_rbac_permissions
  → 0013a_expand_version_num   (bridge, 24 caracteres, cabe en VARCHAR(32))
    → 0014_legacy_caja_role_permissions
      → ... → 0023_physical_counts (head)
```

- La revisión puente se llama `0013a_expand_version_num` (24 caracteres), de modo que puede registrarse en la columna `VARCHAR(32)` antes de ser ampliada.
- En PostgreSQL, `upgrade()` ejecuta `ALTER TABLE alembic_version ALTER COLUMN version_num TYPE VARCHAR(128)`.
- En PostgreSQL, `downgrade()` valida que la revisión actual no exceda 32 caracteres antes de reducir la columna a `VARCHAR(32)`.
- En SQLite, ambas operaciones son no-op, porque SQLite no impone el límite de longitud declarado y la cadena debe permanecer idéntica.

## Corrección posterior a auditoría

El arquitecto detectó un defecto crítico en la primera implementación del bridge. Esta sección documenta el hallazgo y la corrección aplicada en el segundo commit (`test(db): fix PostgreSQL Alembic capacity DDL`). El commit `c3b1259` no fue reescrito: no se usó `amend`, `rebase`, `squash`, `push` ni `merge`.

### Defecto detectado

La primera implementación construía el `ALTER TABLE ... TYPE VARCHAR(...)` con un parámetro enlazado en el modificador de tipo:

```python
sa.text("ALTER TABLE alembic_version ALTER COLUMN version_num TYPE VARCHAR(:length)").bindparams(length=128)
```

SQLAlchemy compila esa expresión, para el dialecto PostgreSQL, como:

```sql
ALTER TABLE alembic_version ALTER COLUMN version_num TYPE VARCHAR(%(length)s)
```

PostgreSQL **no admite un parámetro enlazado en la posición de un modificador de tipo**. El `upgrade` podría fallar con un error de sintaxis al ejecutarse contra la base productiva.

Adicionalmente, las pruebas iniciales sólo inspeccionaban el texto fuente del archivo (`"VARCHAR" in source`, `"128" in source`, `"32" in source`), lo cual era insuficiente: una prueba basada en presencia de palabras no verifica que el DDL resultante sea ejecutable ni que carezca de placeholders.

### Corrección aplicada

- Se sustituyeron ambos `op.execute(sa.text(... VARCHAR(:length)).bindparams(...))` por la API tipada de Alembic:
  - upgrade: `op.alter_column("alembic_version", "version_num", existing_type=sa.String(length=32), type_=sa.String(length=128))`;
  - downgrade: `op.alter_column("alembic_version", "version_num", existing_type=sa.String(length=128), type_=sa.String(length=32))`.
- `sa.String(length=N)` se compila, para el dialecto PostgreSQL, como el literal `VARCHAR(N)` sin placeholders. Se verificó empíricamente que el DDL resultante es `VARCHAR(128)` y `VARCHAR(32)` exactos.
- Se conservan: la ejecución únicamente cuando `bind.dialect.name == "postgresql"`, SQLite como no-op, el guard previo al downgrade y los identificadores de revisión sin cambios.

### Pruebas nuevas

Las pruebas de fuente fueron reemplazadas por pruebas a nivel de operación/compilación:

- `test_upgrade_calls_alter_column_32_to_128`: ejecuta `upgrade()` con un `op` proxy sobre dialecto PostgreSQL, confirma `op.alter_column("alembic_version", "version_num", existing_type=String(32), type_=String(128))` y compila la operación completa para verificar `ALTER TABLE alembic_version ALTER COLUMN version_num TYPE VARCHAR(128);` sin placeholders (`:length`, `%(length)s`, `$1`, `?`).
- `test_downgrade_calls_alter_column_128_to_32_when_revision_fits`: verifica `String(128)` → `String(32)` y compila la operación completa a `ALTER TABLE alembic_version ALTER COLUMN version_num TYPE VARCHAR(32);`.
- `test_downgrade_guard_rejects_revision_longer_than_32`: verifica que el guard lanza `RuntimeError` cuando la revisión actual supera 32 caracteres y que en ese caso no se invoca `op.alter_column`.
- `test_sqlite_branch_does_not_call_alter_column`: verifica que ni `upgrade()` ni `downgrade()` invocan `op.alter_column` cuando el dialecto es SQLite.

Estas pruebas se suman a las existentes (revisiones ≤128, bridge ≤32, cadena `0013 → bridge → 0014`, única head, round-trip SQLite).
