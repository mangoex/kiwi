## Diagnóstico

Al ejecutar `alembic upgrade head` sobre la base PostgreSQL de Easyplain (detenida en `0013_pos_cash_rbac_permissions`), la transacción se revierte antes de registrar la revisión `0014_legacy_caja_role_permissions`:

```text
StringDataRightTruncation:
value too long for type character varying(32)

UPDATE alembic_version
SET version_num='0014_legacy_caja_role_permissions'
```

La tabla técnica `alembic_version.version_num` es `VARCHAR(32)`, pero los identificadores de revisión 0014 a 0018 miden entre 33 y 37 caracteres. La transacción se revierte y la base permanece en `0013_pos_cash_rbac_permissions`.

## Causa

La longitud del identificador de revisión creció con los nombres descriptivos usados desde la revisión 0010, superando los 32 caracteres del `version_num` estándar de Alembic a partir de la 0014. Las revisiones afectadas:

- `0014_legacy_caja_role_permissions` — 33 caracteres
- `0015_business_units_operational_roles` — 37 caracteres
- `0016_customer_directory_snapshots` — 33 caracteres
- `0017_suppliers_purchase_presentations` — 37 caracteres
- `0018_direct_purchases_cash_costing` — 34 caracteres

`0019_advanced_recipes_production` (32) está justo en el límite y cabía. Las revisiones previas a 0014 no superaban 30 caracteres.

## Estado productivo observado en 0013

La base PostgreSQL de Easyplain se encuentra sana en `0013_pos_cash_rbac_permissions` (30 caracteres). La migración fallida fue revertida por PostgreSQL (DDL transaccional), de modo que no quedó estado parcial. El esquema y los datos de negocio están intactos en ese punto.

## Diseño de la migración puente

Se inserta una migración puente entre `0013_pos_cash_rbac_permissions` y `0014_legacy_caja_role_permissions`:

```
0013_pos_cash_rbac_permissions
  → 0013a_expand_version_num   (bridge, 24 caracteres, cabe en VARCHAR(32))
    → 0014_legacy_caja_role_permissions
      → ... → 0023_physical_counts (head)
```

- La revisión puente se llama `0013a_expand_version_num` (24 caracteres), de modo que puede registrarse en la columna `VARCHAR(32)` antes de ser ampliada.
- En PostgreSQL, `upgrade()` ejecuta `ALTER TABLE alembic_version ALTER COLUMN version_num TYPE VARCHAR(128)`.
- En PostgreSQL, `downgrade()` valida que la revisión actual no exceda 32 caracteres antes de reducir la columna a `VARCHAR(32)`.
- En SQLite, ambas operaciones son no-op, porque SQLite no impone el límite de longitud declarado y la cadena debe permanecer idéntica.

## Pruebas

Las pruebas cubren los casos exigidos:

1. inspección de todos los revision IDs y rechazo de longitudes mayores a 128;
2. confirmación de que el bridge ID cabe en 32 caracteres;
3. confirmación de la secuencia `0013 → 0013a → 0014`;
4. ejecución en SQLite temporal de upgrade/downgrade/upgrade (round-trip) desde `0013_pos_cash_rbac_permissions` hasta head;
5. confirmación de head `0023_physical_counts`;
6. verificación estructural del SQL PostgreSQL (`VARCHAR(128)` en upgrade, `VARCHAR(32)` en downgrade);
7. confirmación de que SQLite no emite `ALTER COLUMN` al cruzar el bridge.

## Riesgos

- **Riesgo de regresión en 0014:** mitigado con prueba `test_0014_only_header_changed_not_body`, que confirma que `CAJA_PERMISSION_CODES`, el `INSERT` y el `DELETE` permanecen. El cuerpo de `upgrade()`/`downgrade()` no se tocó.
- **Downgrade shrink:** la rama PostgreSQL de downgrade valida con guard que la revisión actual no exceda 32 caracteres antes de reducir la columna, evitando truncamiento.
- **SQLite vs PostgreSQL:** el bridge es no-op en SQLite. La prueba `test_sqlite_bridge_does_not_alter_column` confirma que no se emite `ALTER` en SQLite, evitando incompatibilidades con el motor de pruebas.
- **No se ejecutó contra la base de Easyplain.** El SQL PostgreSQL quedó probado estructuralmente (regex sobre el fuente) y funcionalmente vía SQLite (round-trip), pero la verificación final en PostgreSQL la hará el operador durante el despliegue.

## Procedimiento de despliegue (Easyplain)

1. Generar un respaldo de la base (snapshot de PostgreSQL o `pg_dump`).
2. En la consola del servicio API:

```bash
cd /app/apps/api
alembic current -v
```

Debe mostrar `0013_pos_cash_rbac_permissions`.

3. Avanzar la cadena completa:

```bash
alembic upgrade head
```

4. Resultado esperado: `0023_physical_counts`.
5. Verificación posterior: `alembic current -v` muestra `0023_physical_counts`; `/health/ready` muestra `postgres: ok`.

## Rollback

Si fuera necesario revertir:

```bash
cd /app/apps/api
alembic downgrade 0013_pos_cash_rbac_permissions
```

La migración puente reduce `version_num` a `VARCHAR(32)` en PostgreSQL sólo cuando la revisión actual cabe (el bridge ID tiene 24 caracteres). En SQLite es no-op.

Si el downgrade no fuera viable, restaurar el respaldo del paso 1.
